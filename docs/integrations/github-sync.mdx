## Modes

Configuration (and global `--sync`) selects how **GitHub Issues** sync behaves when you pass **`sync --with-github`**:

- **`off`** — `sync --with-github` is rejected; use plain **`sync`** to reconcile the data branch over git only.
- **`outbound`** — push hyper-pm changes to issues.
- **`full`** — outbound plus **inbound** comparison and optional **PR timeline** ingestion for eligible tickets.

## Default `sync` (data branch over git)

Plain **`sync`** (no flags) **only** runs `git fetch` / `git merge` / `git push` for the configured data branch (no Octokit, no `GITHUB_TOKEN`). It runs even when **`sync` is `off`** in config. Use **`sync --with-github`** when you also want GitHub Issues mirroring (see modes above).

The legacy **`--git-data`** flag is a no-op; default behavior is already git-only.

## Outbound behavior

- Issue **labels** include `hyper-pm`, `ticket`, plus each ticket label (GitHub’s label length limits apply).
- Ticket **planning** (`priority`, `size`, `estimate`, schedule) is embedded in a **fenced JSON block** after the issue body so **`full`** inbound can round-trip fields conservatively (only keys present in the fence participate — older issues without keys are not mass-cleared).

## Inbound (`full`)

Compares non-reserved labels and the planning JSON fence to the local projection and appends **`GithubInboundUpdate`** when something differs.

After inbound, for tickets **`in_progress`** with linked issues, hyper-pm may load **linked PR timelines** when bodies reference issues with `Refs` / `Closes` / `Fixes` — appending **`GithubPrActivity`** events. This issues **extra GitHub API** calls per linked PR.

## Issue creation vs update

- **Creates** on GitHub only happen for tickets with a valid **epic + story** chain.
- Tickets **without** a story are **skipped** for creation until you attach `--story`.
- Tickets that **already** have a **`githubIssueNumber`** are still **updated** when unlinked (the issue body drops parent ids until you link again).

## Assignees

Tickets store **at most one** assignee (GitHub login). Outbound sync sets that user as the issue’s assignee (additional assignees on the issue may be removed). Inbound reads the **first** assignee when GitHub returns several.

<Callout type="warn">

**`sync --skip-network`** (legacy: **`--no-github`**) skips **all** sync network work (git and GitHub). **`sync --with-github`** requires **`GITHUB_TOKEN`** or **`gh`** and needs **`sync`** not **`off`** in config.

</Callout>
